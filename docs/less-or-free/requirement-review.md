## 问题列表

### 问题 #1 [严重]

- **维度**：遗漏检查
- **位置**：功能需求 > 模块：后台管理
- **描述**：PRD 风险 R-004 提到"管理员后台可删除评论"作为缓解措施，但功能需求中没有评论删除的 REQ。管理员无法处理恶意/垃圾评论。
- **建议修改**：新增 REQ-015：评论管理，包含管理员删除评论功能，删除后 comment_count 同步更新。

### 问题 #2 [严重]

- **维度**：遗漏检查
- **位置**：功能需求 > 模块：后台管理
- **描述**：PRD 中分类（categories）是种子数据初始化的，但没有提供管理员在后台新增/编辑/删除分类的功能。如果需要新增一个分类（如"AI 辅助"），只能手动操作数据库。
- **建议修改**：新增 REQ-016：分类管理，管理员可在后台新增/编辑/删除分类，包含名称、slug、排序权重的编辑。

### 问题 #3 [严重]

- **维度**：遗漏检查
- **位置**：功能需求 > 模块：后台管理
- **描述**：工具只有 draft/published/unpublished 三种状态，没有"永久删除"功能。如果管理员误添加了一个工具，无法彻底删除，只能下架。此外，已删除工具的评论和交互数据如何处理也未说明。
- **建议修改**：在 REQ-010 中增加工具删除功能，明确删除时关联的评论和交互数据的处理策略（级联删除或软删除）。

### 问题 #4 [严重]

- **维度**：可行性
- **位置**：技术栈 > 渲染策略
- **描述**：PRD 声明前台使用 SSG + ISR，但 Cloudflare Pages 对 ISR 的支持有限。`@cloudflare/next-on-pages` 不原生支持 ISR 的 revalidation 时间机制，需要使用 On-Demand Revalidation（通过 API 手动触发）或回退到 CSR。这直接影响 REQ-011 中"发布成功后前台内容更新"的可行性。
- **建议修改**：1) 将渲染策略明确为 SSG + On-Demand Revalidation，管理员发布时通过 `revalidatePath` API 触发页面更新；2) 在 REQ-011 验收标准中明确"发布后通过 On-Demand Revalidation 触发前台页面更新"。

### 问题 #5 [严重]

- **维度**：清晰性
- **位置**：功能需求 > REQ-011 一键发布
- **描述**："待发布变更"的定义不明确。什么算"待发布"？是所有 status=draft 的工具？还是自上次发布以来状态发生过变更的所有工具？如果是前者，一个长期处于 draft 状态的工具是否每次发布都会被包含？如果是后者，如何追踪"自上次发布以来的变更"？
- **建议修改**：明确定义"待发布变更"为：自上次快照生成以来，状态从 draft 变为 published 的工具 + 信息被修改的已发布工具。在数据模型中为 tools 表增加 `last_snapshot_version` 字段，记录工具最后一次被纳入快照的版本号，通过对比当前版本号判断是否有未发布的变更。

### 问题 #6 [一般]

- **维度**：一致性
- **位置**：功能需求 > REQ 编号
- **描述**：REQ 编号从 REQ-006 直接跳到 REQ-008，REQ-007 是"版本快照与 Diff"但出现在后面的模块中。编号顺序与模块顺序不一致，容易造成混淆。
- **建议修改**：按模块顺序重新编号：前台浏览 REQ-001~003，社区互动 REQ-004~007，版本快照 REQ-008，工具详情 REQ-009，后台管理 REQ-010~015。

### 问题 #7 [一般]

- **维度**：一致性
- **位置**：接口设计 > Star / 反对接口
- **描述**：Star 和反对接口在 toggle 取消时返回 code 409，但 409 语义是"冲突"，用于表示 toggle 取消操作不合适。Toggle 是正常的业务操作而非冲突，应统一返回 200，通过 data.starred / data.opposed 字段区分当前状态。
- **建议修改**：Star 和反对接口无论新增还是取消，均返回 code 200，通过 data.starred / data.opposed 布尔值表示当前状态。

### 问题 #8 [一般]

- **维度**：清晰性
- **位置**：功能需求 > REQ-005 评论功能
- **描述**：评论内容没有长度限制。恶意用户可以提交超长评论（如 10MB 文本），导致数据库存储膨胀和页面渲染异常。
- **建议修改**：在 REQ-005 验收标准中增加：评论内容长度限制为 1~500 字符，超出范围前端校验拦截并提示。

### 问题 #9 [一般]

- **维度**：清晰性
- **位置**：功能需求 > REQ-014 工具详情页
- **描述**：工具主页链接的跳转行为未明确。是否在新标签页打开？是否添加 rel="noopener noreferrer" 安全属性？已下架工具的链接是否仍然可点击？
- **建议修改**：在 REQ-014 验收标准中增加：1) 工具链接在新标签页打开，添加 rel="noopener noreferrer"；2) 已下架工具的链接仍可点击。

### 问题 #10 [一般]

- **维度**：清晰性
- **位置**：功能需求 > REQ-008 免登录交互
- **描述**：浏览器指纹方案未明确。使用哪个指纹库（FingerprintJS 开源版/商业版）？指纹粒度如何？如果用户禁用 JavaScript 或使用隐私模式，指纹生成失败时如何处理交互？
- **建议修改**：明确指纹方案：1) 使用 FingerprintJS 开源版（MIT 协议）；2) 指纹生成失败时降级为随机 UUID 存入 localStorage；3) 在验收标准中增加：指纹生成失败时用户仍可正常交互，但清除浏览器数据后交互状态会丢失。

### 问题 #11 [一般]

- **维度**：一致性
- **位置**：功能需求 > REQ-010 工具管理
- **描述**：REQ-010 验收标准第 3 条"编辑已有工具，修改后状态保持不变（已发布的仍为 published）"与"草稿→发布"流程存在逻辑矛盾。如果编辑已发布工具后立即生效，那修改的内容直接暴露给用户，没有经过"待发布"审核环节。这与 REQ-011 的"一键发布"流程不一致。
- **建议修改**：明确编辑已发布工具时的行为策略：方案 A（推荐）— 编辑已发布工具时，修改立即生效但标记为"有未发布变更"，下次发布时生成快照记录；方案 B — 编辑已发布工具时，创建一个 draft 副本，发布后替换原工具。建议采用方案 A，并在 REQ-010 中明确说明。

### 问题 #12 [一般]

- **维度**：可行性
- **位置**：数据模型 > snapshots
- **描述**：快照的 data 字段存储所有已发布工具的完整 JSON。随着工具数量增长（数百个工具 × 完整信息），单个快照可能达到数百 KB 甚至 MB 级别。D1 单行大小限制为 1MB，可能超限。
- **建议修改**：1) 快照 JSON 仅存储必要字段（id, name, badge_*, status, category_ids），不存储 description 和 url 等大字段；2) 或改为快照仅存储增量变更（相对于上一版本的 diff），而非全量数据；3) 在非功能需求中增加快照大小监控指标。

### 问题 #13 [一般]

- **维度**：完整性
- **位置**：功能需求 > REQ-014 工具详情页
- **描述**：已下架工具是否仍允许 star/评论/反对？PRD 未明确。从交互视觉角度，已下架工具的互动区域应该如何展示？
- **建议修改**：在 REQ-014 中明确：已下架工具仍允许 star 和评论（用户可能仍在使用），但反对按钮禁用（已下架无需再反对）。或明确为：已下架工具所有互动按钮均禁用，仅保留浏览和评论查看功能。

### 问题 #14 [一般]

- **维度**：可行性
- **位置**：数据模型 > snapshots
- **描述**：D1 不支持原生的 AUTO INCREMENT。snapshots 表的 version 字段标注为"自增"，需要通过应用层实现（查询最大 version + 1），存在并发竞态风险。
- **建议修改**：在数据模型说明中注明：version 自增通过应用层 `SELECT MAX(version) + 1` 实现，发布操作为单管理员单次操作，不存在并发竞态问题。或使用 UUID 作为主键，version 通过发布时计数器生成。

### 问题 #15 [建议]

- **维度**：遗漏检查
- **位置**：非功能需求
- **描述**：缺少 SEO 相关需求。作为一个收录工具的平台，搜索引擎收录至关重要。缺少 meta tags、Open Graph、sitemap、结构化数据等需求。
- **建议修改**：在非功能需求中增加：NFR-011 SEO — 每个工具详情页包含独立的 title/meta description/Open Graph 标签；自动生成 sitemap.xml；工具详情页添加 JSON-LD 结构化数据（SoftwareApplication 类型）。

### 问题 #16 [建议]

- **维度**：遗漏检查
- **位置**：非功能需求
- **描述**：缺少无障碍（a11y）需求。徽章系统依赖颜色传达信息（绿/黄/红），对色觉障碍用户不友好。
- **建议修改**：在非功能需求中增加：NFR-012 无障碍 — 徽章除颜色外还需通过符号和文字 tooltip 区分含义；所有交互元素支持键盘操作；图片和图标提供 aria-label。

### 问题 #17 [建议]

- **维度**：可测试性
- **位置**：功能需求 > REQ-003 徽章系统
- **描述**："悬停时显示 tooltip"在移动端无法触发（无 hover 事件），验收标准不可测试。
- **建议修改**：修改为：桌面端悬停显示 tooltip，移动端点击/长按显示 tooltip。

### 问题 #18 [建议]

- **维度**：可测试性
- **位置**：功能需求 > REQ-004/006
- **描述**："star 数量实时更新"和"反对数量实时更新"中"实时"未量化。是毫秒级？秒级？还是页面刷新后更新？
- **建议修改**：明确为"点击后前端立即更新显示计数（乐观更新），API 返回后校验修正，误差不超过 1 次 API 往返时间"。

### 问题 #19 [建议]

- **维度**：遗漏检查
- **位置**：数据模型 > tools
- **描述**：工具缺少图标/logo 字段。工具卡片和详情页如果只有文字，视觉上会很单调，用户体验差。
- **建议修改**：为 tools 表增加 `icon_url` 字段（TEXT，选填），存储工具的图标/Logo URL。工具卡片展示图标，无图标时显示首字母头像。

### 问题 #20 [建议]

- **维度**：遗漏检查
- **位置**：功能需求 > REQ-002 工具大全
- **描述**：工具大全的 badge 筛选格式 `badge_payment:free` 不够直观，且未说明如何同时筛选多个徽章条件（如"免费 + 无需翻墙"）。
- **建议修改**：修改筛选参数设计，支持多条件组合：`?badge_payment=free&badge_china_access=direct`，每个徽章组独立参数，多组之间为 AND 关系。

### 问题 #21 [建议]

- **维度**：清晰性
- **位置**：功能需求 > REQ-006 反对功能
- **描述**：反对的二次确认 UI 形式未明确。是模态弹窗？还是内联确认？交互视觉上差异很大。
- **建议修改**：明确为模态弹窗（Modal Dialog），包含确认文案"确定要反对这个工具吗？"和"确认"/"取消"两个按钮。

### 问题 #22 [建议]

- **维度**：清晰性
- **位置**：功能需求 > REQ-001 分类列表浏览
- **描述**：Plan 中提到首页展示"最新上架工具"，但 PRD REQ-001 只提到"分类入口卡片"，缺少首页最新工具展示区域。
- **建议修改**：在 REQ-001 验收标准中增加：首页展示最近上架的 8 个工具（按 created_at 降序），点击可进入工具详情页。

### 问题 #23 [建议]

- **维度**：遗漏检查
- **位置**：功能需求
- **描述**：Plan 原始需求提到收录内容包括"工具、coding plan、平台"，但 PRD 将所有收录内容统一为"工具"，没有区分不同内容类型。coding plan 和平台与独立工具的属性可能不同（如 coding plan 可能没有 url 字段）。
- **建议修改**：为 tools 表增加 `type` 字段（TEXT：tool / plan / platform），工具卡片和详情页根据 type 展示不同的标签。如果当前阶段不区分，至少在数据模型中预留该字段。

### 问题 #24 [建议]

- **维度**：可测试性
- **位置**：功能需求 > REQ-011 一键发布
- **描述**："发布成功后前台内容更新"未量化更新时效。用户发布后多久能在前台看到变化？
- **建议修改**：明确为"发布成功后 30 秒内，通过 On-Demand Revalidation 触发前台页面更新"。

### 问题 #25 [建议]

- **维度**：完整性
- **位置**：接口设计 > 后台接口
- **描述**：后台工具列表接口 `GET /api/admin/tools` 没有分页参数。当工具数量增长后，一次性返回所有工具会导致性能问题。
- **建议修改**：为 `GET /api/admin/tools` 增加 `page` 和 `pageSize` 分页参数，与前台接口保持一致。

### 问题 #26 [建议]

- **维度**：清晰性
- **位置**：功能需求 > REQ-013 工具上下架
- **描述**：验收标准第 3 条"上下架操作需在下次发布时才生成快照记录"表述不清。是指上下架操作立即在前台生效，但快照在下次发布时才记录？还是上下架操作需要等到下次发布才在前台生效？
- **建议修改**：明确为：上下架操作立即在前台生效（分类列表实时更新），但快照在管理员执行"一键发布"时才生成记录。

## 评审结论说明

本次评审发现 **5 个严重问题**、**9 个一般问题**、**12 个建议**。

**严重问题**主要集中在：
1. 遗漏了关键的后台管理功能（评论删除、分类管理、工具删除）
2. Cloudflare Pages 的 ISR 限制未在技术方案中体现，可能影响"发布后前台更新"的核心流程
3. "待发布变更"的定义模糊，直接影响发布流程的实现

**一般问题**主要集中在：
1. 交互语义不一致（409 用于 toggle 操作）
2. 编辑已发布工具与"草稿→发布"流程的逻辑矛盾
3. 评论长度、指纹方案、链接行为等边界条件未明确

**建议**主要集中在：
1. SEO、无障碍、图标等体验优化
2. 移动端 tooltip、实时更新量化等可测试性改进
3. 内容类型区分、分页等扩展性考虑

建议优先解决 5 个严重问题后再进入实施阶段。
